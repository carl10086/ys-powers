# teach-code

通过交互式问答，带你一步步理解任意代码模块的源码。

Start by understanding the project context, then guide the user through questions to deepen understanding of the code.

<HARD-GATE>
Do NOT invoke any implementation skill, write any code, scaffold any project, or take any implementation action until you have presented a code understanding summary and the user has confirmed they've grasped the key concepts.
</HARD-GATE>

<IMPORTANT>
    - 你必须使用中文提问
    - 你必须使用中文写代码理解笔记
</IMPORTANT>

## Anti-Pattern: "代码太简单不需要深入"

每个代码模块都有其复杂性，即使看起来简单的代码也可能有值得注意的设计决策或边缘情况。"简单"代码可以用少量问题带过，但必须保持由浅入深的引导流程。

## Checklist

你必须按顺序完成以下步骤：

1. **探索上下文** — 读取指定路径代码，分析文件结构、核心模块、依赖关系，生成整体概览
2. **提供可视化辅助**（如需要）— 如有复杂调用链或分支逻辑，提供可视化辅助
3. **提出关键理解问题** — 一次一问，根据用户回答逐步深入
4. **生成代码理解笔记** — 整理关键理解点，写成笔记
5. **自我检查笔记完整性** — 确保笔记清晰、完整、无歧义
6. **用户确认** — 让用户确认理解笔记是否满足需求

## 核心原则

**由浅入深**：
- 先建立整体概览，让用户知道这个模块是做什么的
- 再从关键部分入手，提出一个核心问题
- 根据用户回答决定深入方向，避免无关内容干扰

**一次一问**：
- 不要一次抛出多个问题，这会让用户感到压力
- 等用户回答后再问下一个，让用户掌控节奏

**自动判断深度**：
- AI 根据代码复杂度自动决定要问多少问题
- 简单模块可能只需 2-3 个问题
- 复杂模块可能需要更多轮次

**可选项**：
- 如有复杂调用链或分支逻辑，使用 chrome-devtools MCP 提供可视化辅助
- 使用多选题时优先，减少用户组织语言的成本

## 代码理解笔记格式

最终产出代码理解笔记，保存到 `docs/superpowers/specs/YYYY-MM-DD-<模块名>-read.md`：

```markdown
# <模块名> 代码理解

## 模块概览
<简要说明这个模块是做什么的>

## 核心设计思路
<关键的架构/设计决策>

## 关键实现细节
<重要的代码片段或逻辑解释>

## 依赖关系
<与其他模块的关系，如有>
```

## 自我检查

生成笔记后，检查：
1. **完整性**：是否覆盖了用户关注的核心内容？
2. **清晰度**：语言是否简洁易懂？
3. **准确性**：是否有事实错误或歧义？
4. **可操作性**：用户能否基于这份笔记继续工作？

如有不足，先补充再提交用户确认。
