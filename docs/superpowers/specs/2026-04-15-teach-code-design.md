# teach-code 命令设计

## 背景

用户希望创建一个交互式代码理解命令，能够通过 **AI 主动讲解 + 用户确认** 的方式，带你**由浅入深**理解任意代码模块的源码。核心是**深入浅出**——用最简单的语言让用户理解最复杂的逻辑。

核心改进：从"AI 一次性讲完"改为**逐步讲解 + 即时确认**的循环流程。

## 目标

- 用户随意指定文件/目录/模块作为起点
- AI 先宏观再细节，逐层深入
- 每层讲解后主动确认用户理解，再进入下一层
- 最终产出代码理解笔记

## 设计决策

### 核心定位

**从"AI 审问"变为"AI 主动教"**：

| 旧思路 | 新思路 |
|--------|--------|
| AI 一次性讲完所有内容 | 逐步讲解，每步确认 |
| 用户事后追问 | 用户即时确认，可以随时追问 |
| 宏观微观混在一起 | 先宏观再细节，层级清晰 |

### 讲解层级（从宏观到细节）

每一层都有明确的**入口条件和出口条件**：

| 层级 | 回答的问题 | 出口条件 |
|------|-----------|---------|
| **第一层：整体概览** | 这个模块是做什么的？在系统中扮演什么角色？ | 用户理解模块定位 |
| **第二层：核心概念拆分** | 有哪些主要概念？各解决什么问题？ | 用户理解核心概念 |
| **第三层：关键设计决策** | 为什么这样设计？有什么替代方案？ | 用户理解设计思路 |
| **第四层：实现细节** | 具体怎么实现的？边界情况？ | 用户要求深入时 |

### 核心流程

```
Step 1: 整体概览
  → 生成"模块定位"和"在系统中的位置"
  → 用户确认宏观理解后，进入下一步

Step 2: 核心概念拆分
  → 概念1 → 讲解1 → 互动确认
  → 概念2 → 讲解2 → 互动确认
  → 用户理解核心概念后，进入下一步

Step 3: 关键设计决策
  → 设计点1 → 讲解 + 互动
  → 设计点2 → 讲解 + 互动

Step 4: 实现细节（可选）
  → 用户主动要求深入时才讲

Step 5: 总结
  → 生成代码理解笔记
  → 用户确认
```

### 互动机制

**AI 何时主动提问**：

| 信号类型 | 说明 |
|---------|------|
| **概念边界** | 一个逻辑单元讲完了 |
| **关键设计点** | 遇到了重要的设计决策 |
| **复杂度升级** | 从简单进入复杂 |
| **用户困惑风险** | 术语或抽象层级跳跃大 |

**问题形式**（AI 自动选择）：
- 理解确认、设计追问、细节深挖、关系串联、应用场景

### 深入浅出的技巧

- **类比**：用熟悉的事物比喻陌生的概念
- **拆解**：把复杂逻辑拆成简单步骤
- **强调**：标记关键点，让用户抓住重点

### 文件结构

```
ys-powers/
├── commands/
│   └── teach-code.md         # command 文件
└── docs/superpowers/specs/
    └── YYYY-MM-DD-teach-code-design.md  # 本设计文档
```

## 验证标准

- `commands/teach-code.md` 存在
- 明确说明"先宏观再细节"的讲解层级
- 讲解流程包含：整体概览 → 核心概念 → 设计决策 → 实现细节
- 互动机制清晰：AI 自动判断何时确认 + 用户随时追问
