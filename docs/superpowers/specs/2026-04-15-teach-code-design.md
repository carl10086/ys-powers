# teach-code 命令设计

## 背景

用户希望创建一个交互式代码理解命令，能够通过 **AI 逐步讲解 + 即时互动** 的方式，带你由浅入深理解任意代码模块的源码。

核心改进：从"一次性讲完"改为**讲一段、问一段**的循环流程。

## 目标

- 先宏观再细节，逐层深入
- 每段讲解后必须停下来问问题
- 问题要引发思考，不是确认回忆
- 用户主导节奏，随时可以追问或跳过
- 最终产出代码理解笔记

## 核心节奏

```
讲解1（一个核心概念）
    ↓
互动1（引发思考的问题）
    ↓
讲解2（下一个概念）
    ↓
互动2
    ↓
...直到讲完
    ↓
总结
```

## 讲解层级

| 层级 | 内容 | 何时进入 |
|------|------|---------|
| 模块概览 | 这个模块做什么的？在系统中什么角色？ | 开始 |
| 核心概念 | 拆成几个主要概念，逐个讲清楚 | 概览完成后 |
| 关键设计 | 为什么这样设计？替代方案？ | 概念理解后 |
| 实现细节 | 具体数据结构、核心函数逻辑 | 用户主动要求时 |

## 互动规则

### 每段讲解后必须问问题

讲完一个完整的意思后，必须停下来问一个问题。

### 问题质量标准

好的问题引发思考：
- "为什么要用这个设计？"
- "这和刚才讲的 X 有什么关系？"
- "如果参数是空的话会怎样？"
- "你有没有想到其他实现方式？"

坏的问题确认回忆：
- "你理解了吗"
- "记住了吗"

### 用户主导

- 随时追问细节
- 说"我懂了，继续"
- 要求深入某个点
- 跳过不关心的内容

## 文件结构

```
ys-powers/
├── commands/
│   └── teach-code.md         # command 文件
└── docs/superpowers/specs/
    └── YYYY-MM-DD-teach-code-design.md
```

## 验证标准

- `commands/teach-code.md` 存在
- 核心节奏清晰：讲一段问一段
- 问题质量有指导
- 用户主导权明确
